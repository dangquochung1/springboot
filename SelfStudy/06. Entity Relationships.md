# Entity Relationships (Mối quan hệ giữa các thực thể)

<br />

## 1. Bản chất của quan hệ One-To-Many (1-N)
Trong thực tế, một Chuyên ngành (Major) sẽ quản lý rất nhiều Sinh viên (Student). 
- Database: Bảng Sinh viên có một cột khóa ngoại (Foreign Key - FK) trỏ về Primary Key của bảng Chuyên ngành.
- Lập trình OOP: Object Major sẽ sở hữu một Danh sách (`List<Student>`) để chứa các sinh viên của mình.

<br />

## 2. Ánh xạ quan hệ 1-N trong JPA

### Ví dụ: Chuyên ngành (1) - Sinh viên (N)

**Bảng "Bên 1" (Major)**
```java
@Entity
@Table(name="Major")
public class Major {
    
    @Id
    @Column(name="id", columnDefinition = "char(2)")
    private String id; // VD: SE

    @Column(name = "name")
    private String name; // VD: Software Engineering

    // Cấu hình Mapping 1-N
    // @JoinColumn: Chỉ định tên cột Khóa ngoại sẽ được TỰ ĐỘNG SINH RA bên bảng Nhiều (Student)
    @OneToMany(cascade = CascadeType.ALL, fetch = FetchType.EAGER)
    @JoinColumn(name="MajorId") 
    private List<Student> students = new ArrayList<>();

    // Hàm tiện ích để thêm sinh viên vào danh sách
    public void addStudent(Student obj){
        students.add(obj);
    }
}
```

**Bảng "Bên N" (Student)**
Bảng Student vẫn khai báo các field bình thường, không cần khai báo tường minh cột Khóa Ngoại `MajorId` vì JPA sẽ tự tạo cột đó khi đọc cấu hình `@JoinColumn` từ Major.

```java
@Entity
@Table(name="Student")
public class Student {
    
    @Id
    @Column(name = "id", columnDefinition = "char(8)")
    private String id; // VD: SE123456
    
    @Column(name = "name", nullable = false)
    private String name;
    
    @Column(name = "yob")
    private int yob;
}
```

## 3. Các thuộc tính quan trọng trong @OneToMany

### 3.1. Cascade (Hiệu ứng Domino)
`cascade = CascadeType.ALL`
Khi lưu (persist) hoặc xóa (remove) thực thể "Bên 1" (Major), thì **toàn bộ** các thực thể "Bên N" (các Student trong danh sách) cũng sẽ bị lưu/xóa theo hiệu ứng dây chuyền (domino).

```java
public static void createMajorStudents() {
    Major seMajor = new Major("SE", "Software Engineering");
    Student s1 = new Student("SE100", "An Nguyen", 2006);
    Student s2 = new Student("SE101", "Binh Nguyen", 2006);

    // Major ôm lấy 2 sinh viên
    seMajor.addStudent(s1);
    seMajor.addStudent(s2);

    EntityManager em = JpaUtil.getEntityManager();
    em.getTransaction().begin();
    
    // NHỜ CÓ CASCADE: Ta CHỈ CẦN PERSIST MAJOR.
    // Major đi xuống database, 2 sinh viên tự động trôi xuống theo vào bảng Student, có gắn sẵn MajorId="SE"!
    em.persist(seMajor); 
    
    em.getTransaction().commit();
}
```

### 3.2. FetchType (Cách tải dữ liệu)
Là cách cấu hình xem khi gọi DB để lấy thông tin Major lên, thì list Student đi kèm có được lấy lên CÙNG LÚC hay không.

- `FetchType.LAZY` (Mặc định cho danh sách): **Lười biếng**. Khi gọi `find(Major, "SE")`, JPA chỉ sinh SQL lấy đúng bảng Major lên. Khi nào bạn thực sự gọi `seMajor.getStudents().size()` thì nó mới tiếp tục sinh ra câu lệnh SQL thứ 2 lấy sinh viên. Giúp tiết kiệm RAM nếu bạn chỉ cần xem tên chuyên ngành.
- `FetchType.EAGER`: **Sốt sắng**. Khi gọi `find(Major, "SE")`, JPA lập tức dùng lệnh `JOIN` để lôi tuột cả 10.000 sinh viên của ngành đó bỏ vào RAM luôn, dù bạn có cần dùng đến list sinh viên hay không. 

> **Lưu ý**: Hãy cẩn thận với FetchType.EAGER trên danh sách OneToMany vì nó có thể làm tràn RAM hệ thống nếu lượng dữ liệu ở "bên N" quá lớn!
