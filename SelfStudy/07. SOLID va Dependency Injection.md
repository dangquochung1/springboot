# SOLID và Dependency Injection (DI)

<br />

## 1. SOLID Principles (Nguyên tắc SOLID)

SOLID là 5 nguyên tắc thiết kế phần mềm linh hoạt, dễ bảo trì, dễ nâng cấp trong lập trình Hướng Đối Tượng (OOP). Nhấn mạnh vào 2 chữ S và O trong bộ thực hành này:

- **S - Single Responsibility Principle (SRP):** Một class chỉ nên giữ 1 trách nhiệm duy nhất, và chỉ có 1 lý do để thay đổi.
  *(Ví dụ: `UserService` chỉ xử lý logic User, `EmailSender` chỉ chuyên gửi email. Không nên nhét code gửi email vào trong UserService).*
  
- **O - Open/Closed Principle (OCP):** Một module/class nên MỞ (Open) cho sự mở rộng, nhưng ĐÓNG (Closed) cho sự sửa đổi (không được sửa code gốc).
  *(Ví dụ: Tương lai muốn gửi tin nhắn qua Zalo/WhatsApp, không nên chui vào `UserService` sửa lại code cũ, mà nên thiết kế sao cho chỉ cần tạo class mới cắm vào hệ thống).*

<br />

## 2. Tight Coupling (Kết nối chặt chẽ)
Là tình huống mà một Class A "phụ thuộc cứng" vào Class B. Class A tự dùng từ khóa `new` để tạo ra Object B bên trong lòng nó.

```java
public class UserService {
    // TỰ NEW ở đây gọi là Full-control, Direct-control
    // Bị "cứng ngắc" (hard-coded dependency) vào đúng 1 class EmailSender
    private EmailSender emailSender = new EmailSender(); 

    public void registerAccount(Account acc) {
        // ...
        emailSender.sendEmail("test@email.com", "Mã OTP");
    }
}
```
**Vấn đề**: Sau này sếp yêu cầu gửi OTP bằng WhatsApp chứ không phải Email. Developer BẮT BUỘC phải vào code `UserService` xóa `new EmailSender()` và sửa thành `new WhatsAppSender()`. Vi phạm nguyên tắc Open/Closed!

## 3. Loose Coupling (Kết nối lỏng lẻo) qua Dependency Injection

### Bước 1: Trừu tượng hóa với Interface
Để lỏng lẻo, `UserService` không nên biết cụ thể ai là người gửi (Email hay WhatsApp). Nó chỉ cần biết: "Cho tôi một ai đó biết `sendNoti()` là được". 
--> Giải quyết bằng Interface.

```java
public interface NotiService {
    void sendNoti(String to, String message);
}

// Các class cụ thể (Impl)
public class EmailSender implements NotiService { ... }
public class WhatsAppSender implements NotiService { ... }
```

### Bước 2: Bỏ từ khóa "new", dùng DI (Dependency Injection)
- **Dependency**: Sự phụ thuộc (`UserService` phụ thuộc vào `NotiService`).
- **Injection**: Tiêm, chích. (Tiêm từ bên ngoài vào, chứ bên trong không tự sinh ra).

`UserService` khai báo Interface, **KHÔNG CÓ TỪ KHÓA NEW**. Nó sẽ yêu cầu bên ngoài tiêm (truyền) obj cụ thể vào thông qua **Constructor** (Khuyên dùng) hoặc **Setter**.

```java
public class UserService {

    // Khai báo chung chung, không NEW!
    private NotiService noti;

    // TIÊM (Injection) qua Constructor
    public UserService(NotiService noti) {
        this.noti = noti;
    }

    public void registerAccount(String to, String msg) {
        noti.sendNoti(to, msg); // Gọi hàm chung chung
    }
}
```

### Bước 3: Tiêm từ bên ngoài (Tầng điều khiển / Main)
Tầng ngoài cùng sẽ tạo obj và quyết định tiêm ai vào cho `UserService` chạy.

```java
public class Main {
    public static void main(String[] args) {
        
        // Cần WhatsApp? Sếp bảo thế, ok. Tạo WhatsApp
        NotiService sender = new WhatsAppSender();
        
        // TIÊM (Chích) nó vào cho Service xài
        UserService userService = new UserService(sender);
        
        userService.registerAccount("098xxx", "Mã OTP của bạn là...");
    }
}
```

**Lợi ích khổng lồ**: 
- Mai mốt có thêm tính năng gửi qua Zalo (`ZaloSender implements NotiService`), `UserService` **KHÔNG BỊ SỬA BẤT CỨ 1 DÒNG CODE NÀO** (Thỏa mãn OCP - Đóng với sự sửa đổi). Chỉ cần ở Main ta tiêm cái ZaloSender vào là xong.
- Code dễ test (dễ Mock).
