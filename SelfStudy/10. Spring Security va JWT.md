# Spring Security và JWT

<br />

## 1. Authentication và Authorization

- **Authentication (Xác thực)**: Trả lời câu hỏi *"Bạn là ai?"*. (Ví dụ: Đăng nhập bằng Username/Password, quét vân tay, dùng Token...).
- **Authorization (Phân quyền)**: Trả lời câu hỏi *"Bạn được phép làm gì?"*. Hệ thống sẽ kiểm tra Vai trò (Roles) hoặc Quyền (Permissions) của bạn có được phép chui vào các Endpoint cụ thể (như GET /users, POST /products) hay không.

<br />

## 2. JWT (JSON Web Token) là gì?

REST API theo chuẩn là **Stateless** (Phi trạng thái). Server không nhớ người dùng là ai sau mỗi request (Không dùng Session). Vậy làm sao server biết ai đang gọi API?

=> Sử dụng **JWT**.
1. Client gửi `username/password` lên Server.
2. Server xác thực đúng, chế ra một chuỗi vé (Token JWT) mã hóa, trong vé chứa thông tin user (`id`, `username`, `roles`...). Trả vé về cho Client.
3. Ở những Request sau, Client kẹp cái vé JWT này vào Header HTTP (`Authorization: Bearer <token>`).
4. Server đọc vé, xác thực chữ ký xem vé có hợp lệ không. Nếu vé thật, trích xuất `roles` ra để kiểm tra quyền hạn trước khi cho chui vào Controller.

<br />

## 3. Kiến trúc Spring Security Filter Chain

Mọi HTTP Request từ Client không bay thẳng vào Controller, mà phải đi qua một "trạm kiểm soát" gọi là **Filter Chain** của Spring Security.

**Luồng chạy (Flow) điển hình:**
```
Client Request + JWT Token
       │
       ▼
[JwtAuthenticationFilter] (Filter do mình tự viết)
  1. Chặn request lại. Lấy Header. Extract Token.
  2. Validate Token hợp lệ.
  3. Lấy ra Username và Roles từ token.
  4. Nạp nó vào `SecurityContextHolder` (Khai báo cho Security biết thằng này đã chứng minh nhân thân xong, và nó là ADMIN).
       │
       ▼
[AuthorizationFilter] (Filter có sẵn của Spring)
  1. Kiểm tra SecurityContext. Nhìn thấy Roles = ADMIN.
  2. Tra bảng luật cấu hình ở `SecurityConfig`.
     - Quy định: `POST /api/fruits` yêu cầu `hasRole("ADMIN")`
  3. Khớp luật! Cho qua / Không khớp luật! Văng 403 Forbidden.
       │
       ▼
[RestController] (Code xử lý DB của mình)
```

## 4. Các cấu hình Code cốt lõi

### 4.1. SecurityConfig (Thiết lập luật chơi)
Class này quy định rành mạch Endpoint nào được thả tự do (Public), Endpoint nào cần Token, và Endpoint nào cần quyền gì (Admin/Member...).

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    // Inject cái Filter bắt JWT của mình làm
    private final JwtAuthenticationFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            // 1. Tắt CSRF vì làm API (Stateless)
            .csrf(csrf -> csrf.disable())

            // 2. Luật chơi (Authorize Requests)
            .authorizeHttpRequests(auth -> auth
                // Endpoint thả rông (Login, Xem danh sách)
                .requestMatchers("/api/auth/login", "/api/auth/register").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/fruits/**").permitAll()
                
                // Phân quyền chuyên sâu
                .requestMatchers(HttpMethod.POST, "/api/fruits").hasRole("ADMIN")
                .requestMatchers(HttpMethod.DELETE, "/api/fruits/**").hasRole("ADMIN")
                
                // Mọi thứ khác: Không cần phân role nhưng bắt buộc phải có vé Token (đã login)
                .anyRequest().authenticated()
            )

            // 3. Khai báo chế độ Stateless
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            
            // 4. Nhét Filter JWT của mình Đứng Lên Trước Filter mặc định của Spring
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

### 4.2. JwtAuthenticationFilter (Cục chặn bắt Token)

Class này kế thừa `OncePerRequestFilter`, đảm bảo mỗi Request chạy vô server đều bị nó quét qua 1 lần. 

Nhiệm vụ: Extract Role từ token -> Nạp vô SecurityContext.

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain) throws ServletException, IOException {
        
        final String authHeader = request.getHeader("Authorization");

        // Nếu Request không chứa vé, hoặc không bắt đầu bằng chữ Bearer -> Cho đi tiếp để bị AuthorizationFilter từ chối sau
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        final String jwt = authHeader.substring(7); // Cắt bỏ 7 chữ "Bearer "
        
        // Gọi thư viện JwtService để lấy Username và Danh sách Role từ trong chuỗi token
        final String username = jwtService.extractUsername(jwt);
        List<GrantedAuthority> authorities = jwtService.extractAuthorities(jwt);

        // Nạp vào SecurityContext của ứng dụng (Báo cho hệ thống biết thằng này đã Login và có Quyền X,Y,Z)
        UsernamePasswordAuthenticationToken authToken = new UsernamePasswordAuthenticationToken(
                username, null, authorities
        );
        SecurityContextHolder.getContext().setAuthentication(authToken);

        // Nạp xong vé thì dở barrier cho đi qua trạm
        filterChain.doFilter(request, response);
    }
}
```

## 5. Tóm lược Checklist Implement Phân quyền
1. Thêm cột `roles` vào Entity User trong Database.
2. Xử lý `JwtService`: Thêm hàm tạo token kẹp roles vào, hàm giải mã lấy roles ra.
3. Xử lý `JwtAuthenticationFilter`: Extract roles từ Token ném vào `SecurityContext`.
4. Cấu hình `SecurityConfig`: Dùng `hasRole()` hoặc `hasAnyRole()` cho các Endpoint bảo mật.
5. Kiểm tra Role trong Controller: Lúc Login, móc roles từ CSDL -> Tạo token.

> **Trọng tâm cực kỳ quan trọng**: Authentication (Filter mình tự viết) nạp Roles. Authorization (Spring làm tự động) check xem Roles nạp đó có đáp ứng Endpoint không.
