## 1. Index — Chỉ mục

### 1.1. Index là gì?

__Index__ (Chỉ mục) là cấu trúc dữ liệu giúp __tăng tốc độ truy vấn__ bằng cách tạo một bản đồ tra cứu nhanh đến vị trí dữ liệu trong bảng, tương tự như mục lục của một cuốn sách.

Không có index, database phải __quét toàn bộ bảng__ (Full Table Scan) để tìm dữ liệu → rất chậm với bảng lớn.

<br />

### 1.2. Các loại Index

| Loại | Mô tả | Khi nào dùng |
| ---- | ----- | ------------ |
| __Clustered Index__ | Sắp xếp __vật lý__ dữ liệu trong bảng. Mỗi bảng chỉ có __1__. | Primary Key (tự động tạo) |
| __Non-Clustered Index__ | Tạo bản đồ tra cứu riêng, __không__ thay đổi thứ tự vật lý. Mỗi bảng có __nhiều__. | Cột thường dùng trong WHERE, JOIN |
| __Unique Index__ | Đảm bảo giá trị không trùng lặp | Email, username, mã nhân viên |
| __Composite Index__ | Index trên __nhiều cột__ | WHERE trên tổ hợp cột |
| __Full-text Index__ | Tìm kiếm văn bản toàn văn | Tìm kiếm nội dung bài viết |
| __Covering Index__ | Index chứa tất cả cột cần truy vấn | SELECT chỉ đọc từ index |

```sql
-- Tạo Non-Clustered Index
CREATE INDEX idx_employee_email ON employees(email);

-- Tạo Unique Index
CREATE UNIQUE INDEX idx_employee_code ON employees(employee_code);

-- Tạo Composite Index
CREATE INDEX idx_dept_name ON employees(department, name);

-- Xóa Index
DROP INDEX idx_employee_email ON employees;
```

<br />

### 1.3. Clustered vs Non-Clustered Index

| Tiêu chí | Clustered Index | Non-Clustered Index |
| -------- | --------------- | ------------------- |
| __Số lượng__ | Chỉ __1__ per table | __Nhiều__ per table |
| __Lưu trữ__ | Sắp xếp lại dữ liệu vật lý | Tạo cấu trúc riêng trỏ đến dữ liệu |
| __Tốc độ đọc__ | Nhanh hơn cho range queries | Nhanh cho lookup cụ thể |
| __Tốc độ ghi__ | Chậm hơn (phải duy trì thứ tự) | Nhanh hơn |
| __Khi nào dùng__ | Primary Key, cột sắp xếp chính | Cột tìm kiếm, filter thường xuyên |

<br />

### 1.4. Khi nào __nên__ và __không nên__ dùng Index

__Nên tạo index khi:__
- Cột thường xuất hiện trong `WHERE`, `JOIN`, `ORDER BY`.
- Bảng có __nhiều dữ liệu__ (hàng trăm nghìn hàng trở lên).
- Cột có __tính chọn lọc cao__ (cardinality cao — nhiều giá trị khác nhau).

__Không nên tạo index khi:__
- Bảng nhỏ (vài trăm hàng).
- Cột có __ít giá trị phân biệt__ (ví dụ: cột `gender` chỉ có M/F).
- Bảng __thường xuyên INSERT/UPDATE/DELETE__ (index làm chậm ghi).
- Cột hiếm khi xuất hiện trong query.

> __Trade-off__: Index __tăng tốc đọc__ nhưng __làm chậm ghi__ (INSERT/UPDATE/DELETE phải cập nhật index). Cần cân nhắc kỹ.

## 2. View

### 2.1. View là gì?

__View__ là một __bảng ảo__ (virtual table) được tạo từ một câu lệnh SELECT. View không lưu trữ dữ liệu thực, mà lưu trữ __câu query__. Mỗi lần truy vấn View, database thực thi query gốc.

```sql
-- Tạo View
CREATE VIEW v_active_employees AS
SELECT e.id, e.name, e.email, d.name AS department
FROM employees e
JOIN departments d ON e.department_id = d.id
WHERE e.status = 'ACTIVE';

-- Sử dụng View (như bảng thường)
SELECT * FROM v_active_employees WHERE department = 'Engineering';

-- Xóa View
DROP VIEW v_active_employees;
```

<br />

### 2.2. Ưu / Nhược điểm

| Ưu điểm | Nhược điểm |
| -------- | ---------- |
| Đơn giản hóa query phức tạp | Có thể chậm (chạy query mỗi lần truy cập) |
| Bảo mật: ẩn cột nhạy cảm | Không phải lúc nào cũng UPDATE/INSERT được |
| Tái sử dụng logic query | Phụ thuộc vào bảng gốc |
| Trừu tượng hóa cấu trúc database | — |

<br />

### 2.3. Materialized View

__Materialized View__ lưu trữ __kết quả thực__ (không phải chỉ câu query), cần __refresh__ để cập nhật.

| Tiêu chí | View | Materialized View |
| -------- | ---- | ----------------- |
| __Lưu trữ__ | Chỉ lưu câu query | Lưu dữ liệu kết quả |
| __Tốc độ__ | Chậm (chạy query mỗi lần) | Nhanh (đọc từ cache) |
| __Cập nhật__ | Luôn mới nhất | Cần refresh thủ công/định kỳ |
| __Phù hợp__ | Query nhẹ, dữ liệu thay đổi thường xuyên | Query nặng, dữ liệu ít thay đổi |

```sql
-- PostgreSQL: Tạo Materialized View
CREATE MATERIALIZED VIEW mv_department_stats AS
SELECT department, COUNT(*) AS total, AVG(salary) AS avg_salary
FROM employees
GROUP BY department;

-- Refresh khi cần cập nhật
REFRESH MATERIALIZED VIEW mv_department_stats;
```

## 3. Normalization — Chuẩn hóa

### 3.1. Chuẩn hóa là gì?

__Normalization__ (Chuẩn hóa) là quá trình tổ chức dữ liệu trong database để __giảm thiểu dư thừa__ (redundancy) và __đảm bảo tính toàn vẹn__ (data integrity).

<br />

### 3.2. Các dạng chuẩn

| Dạng chuẩn | Quy tắc | Ví dụ vi phạm |
| ---------- | ------- | ------------- |
| __1NF__ | Mỗi cột chỉ chứa __giá trị nguyên tử__ (atomic). Không có cột chứa danh sách. | Cột `phones` chứa "0901, 0902" |
| __2NF__ | Đạt 1NF + Mọi cột __phụ thuộc hoàn toàn__ vào toàn bộ Primary Key (không phụ thuộc vào 1 phần PK). | Bảng `(student_id, course_id, student_name)` — `student_name` chỉ phụ thuộc `student_id` |
| __3NF__ | Đạt 2NF + __Không có phụ thuộc bắc cầu__ (transitive dependency). Mọi cột phụ thuộc trực tiếp vào PK. | `(emp_id, dept_id, dept_name)` — `dept_name` phụ thuộc vào `dept_id`, không phải `emp_id` |
| __BCNF__ | Dạng mạnh hơn 3NF. Mọi phụ thuộc hàm `X → Y`, X phải là __superkey__. | Trường hợp hiếm khi PK tạo anomaly |

<br />

### 3.3. Ví dụ chuẩn hóa

__Chưa chuẩn hóa:__
| emp_id | emp_name | dept_id | dept_name | dept_location |
| ------ | -------- | ------- | --------- | ------------- |
| 1 | An | D1 | Engineering | Floor 5 |
| 2 | Binh | D1 | Engineering | Floor 5 |
| 3 | Cuong | D2 | Marketing | Floor 3 |

→ Dư thừa: `dept_name` và `dept_location` lặp lại.

__Sau 3NF:__

__employees__
| emp_id | emp_name | dept_id |
| ------ | -------- | ------- |
| 1 | An | D1 |
| 2 | Binh | D1 |
| 3 | Cuong | D2 |

__departments__
| dept_id | dept_name | dept_location |
| ------- | --------- | ------------- |
| D1 | Engineering | Floor 5 |
| D2 | Marketing | Floor 3 |

<br />

### 3.4. Normalization vs Denormalization

| Tiêu chí | Normalization | Denormalization |
| -------- | ------------- | --------------- |
| __Mục đích__ | Giảm dư thừa, đảm bảo toàn vẹn | Tăng tốc đọc |
| __Dữ liệu__ | Không trùng lặp | Có thể trùng lặp |
| __Số bảng__ | Nhiều bảng nhỏ | Ít bảng lớn |
| __Đọc__ | Cần JOIN → có thể chậm | Ít JOIN → nhanh |
| __Ghi__ | Nhanh (chỉ update 1 nơi) | Chậm (update nhiều nơi) |
| __Dùng cho__ | OLTP (giao dịch) | OLAP (phân tích), Data Warehouse |

## 4. Constraint — Ràng buộc

### 4.1. Các loại Constraint

| Constraint | Mô tả |
| ---------- | ----- |
| `PRIMARY KEY` | Xác định khóa chính (duy nhất + NOT NULL) |
| `FOREIGN KEY` | Tạo quan hệ giữa 2 bảng, đảm bảo toàn vẹn tham chiếu |
| `UNIQUE` | Giá trị không trùng lặp (cho phép NULL) |
| `NOT NULL` | Không cho phép giá trị NULL |
| `CHECK` | Kiểm tra điều kiện logic |
| `DEFAULT` | Giá trị mặc định khi không truyền vào |

```sql
CREATE TABLE employees (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(150) UNIQUE,
    age INT CHECK (age >= 18 AND age <= 65),
    salary DECIMAL(10,2) DEFAULT 0.00,
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(id)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);
```

<br />

### 4.2. Referential Actions (ON DELETE / ON UPDATE)

| Action | Mô tả |
| ------ | ----- |
| `CASCADE` | Xóa/sửa bản ghi cha → tự động xóa/sửa bản ghi con |
| `SET NULL` | Xóa bản ghi cha → FK ở bản ghi con thành NULL |
| `RESTRICT` | Ngăn xóa/sửa bản ghi cha nếu có bản ghi con tham chiếu |
| `NO ACTION` | Tương tự RESTRICT (kiểm tra cuối transaction) |
| `SET DEFAULT` | Đặt FK về giá trị mặc định |

## 5. Stored Procedure và Function

### 5.1. Stored Procedure

__Stored Procedure__ là một nhóm câu lệnh SQL được __biên dịch và lưu trữ__ trong database, có thể gọi lại bất kỳ lúc nào.

```sql
-- MySQL: Tạo Stored Procedure
DELIMITER //
CREATE PROCEDURE GetEmployeesByDepartment(IN dept_name VARCHAR(50))
BEGIN
    SELECT e.id, e.name, e.email, e.salary
    FROM employees e
    JOIN departments d ON e.department_id = d.id
    WHERE d.name = dept_name
    ORDER BY e.salary DESC;
END //
DELIMITER ;

-- Gọi Stored Procedure
CALL GetEmployeesByDepartment('Engineering');
```

```sql
-- Stored Procedure với OUTPUT parameter
DELIMITER //
CREATE PROCEDURE GetDepartmentStats(
    IN dept_name VARCHAR(50),
    OUT total_emp INT,
    OUT avg_salary DECIMAL(10,2)
)
BEGIN
    SELECT COUNT(*), AVG(salary)
    INTO total_emp, avg_salary
    FROM employees e
    JOIN departments d ON e.department_id = d.id
    WHERE d.name = dept_name;
END //
DELIMITER ;

-- Gọi và lấy kết quả
CALL GetDepartmentStats('Engineering', @total, @avg);
SELECT @total AS total_employees, @avg AS average_salary;
```

<br />

### 5.2. User-Defined Function (UDF)

__Function__ luôn __trả về một giá trị__ và có thể dùng trong câu SELECT, WHERE.

```sql
-- Tạo Function
DELIMITER //
CREATE FUNCTION CalculateBonus(salary DECIMAL(10,2), performance INT)
RETURNS DECIMAL(10,2)
DETERMINISTIC
BEGIN
    DECLARE bonus DECIMAL(10,2);
    IF performance >= 9 THEN
        SET bonus = salary * 0.20;
    ELSEIF performance >= 7 THEN
        SET bonus = salary * 0.10;
    ELSE
        SET bonus = salary * 0.05;
    END IF;
    RETURN bonus;
END //
DELIMITER ;

-- Sử dụng trong SELECT
SELECT name, salary, CalculateBonus(salary, performance_score) AS bonus
FROM employees;
```

<br />

### 5.3. Stored Procedure vs Function

| Tiêu chí | Stored Procedure | Function |
| -------- | ---------------- | -------- |
| __Giá trị trả về__ | Không bắt buộc (có thể nhiều output) | Luôn trả về __1 giá trị__ |
| __Dùng trong SELECT__ | ❌ Không | ✅ Có |
| __Transaction__ | ✅ Hỗ trợ | ❌ Không (hầu hết RDBMS) |
| __Side effects__ | Có thể INSERT/UPDATE/DELETE | Chỉ nên tính toán, không thay đổi dữ liệu |
| __Gọi__ | `CALL procedure_name()` | Dùng như hàm built-in |

## 6. CTE — Common Table Expression

### 6.1. CTE là gì?

__CTE (Common Table Expression)__ là một __tập kết quả tạm thời__ có tên, được định nghĩa bằng `WITH`. CTE giúp viết query phức tạp __dễ đọc__ và __dễ bảo trì__ hơn.

```sql
-- CTE đơn giản
WITH department_stats AS (
    SELECT department_id, COUNT(*) AS total, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department_id
)
SELECT d.name, ds.total, ds.avg_salary
FROM department_stats ds
JOIN departments d ON ds.department_id = d.id
WHERE ds.avg_salary > 15000000;
```

<br />

### 6.2. Recursive CTE

__Recursive CTE__ cho phép tham chiếu lại chính nó, phù hợp cho __dữ liệu phân cấp__ (cây tổ chức, danh mục).

```sql
-- Tìm tất cả quản lý cấp trên của nhân viên
WITH RECURSIVE management_chain AS (
    -- Base case: nhân viên ban đầu
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE id = 10

    UNION ALL

    -- Recursive: lên cấp quản lý tiếp theo
    SELECT e.id, e.name, e.manager_id, mc.level + 1
    FROM employees e
    JOIN management_chain mc ON e.id = mc.manager_id
)
SELECT * FROM management_chain ORDER BY level;
```

<br />

### 6.3. CTE vs Subquery

| Tiêu chí | CTE | Subquery |
| -------- | --- | -------- |
| __Đọc hiểu__ | Dễ đọc, có tên rõ ràng | Khó đọc khi lồng sâu |
| __Tái sử dụng__ | Dùng lại nhiều lần trong query | Phải viết lại nếu dùng lại |
| __Recursive__ | ✅ Hỗ trợ | ❌ Không |
| __Performance__ | Tương đương (optimizer thường xử lý giống nhau) | Tương đương |

## 7. Trigger

### 7.1. Trigger là gì?

__Trigger__ là một đoạn code SQL __tự động thực thi__ khi có sự kiện xảy ra trên bảng (INSERT, UPDATE, DELETE).

```sql
-- Tạo Trigger: Tự động log khi salary thay đổi
DELIMITER //
CREATE TRIGGER trg_salary_change
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    IF OLD.salary <> NEW.salary THEN
        INSERT INTO salary_audit_log (employee_id, old_salary, new_salary, changed_at)
        VALUES (NEW.id, OLD.salary, NEW.salary, NOW());
    END IF;
END //
DELIMITER ;
```

| Loại | Mô tả |
| ---- | ----- |
| `BEFORE INSERT` | Chạy trước khi INSERT |
| `AFTER INSERT` | Chạy sau khi INSERT |
| `BEFORE UPDATE` | Chạy trước khi UPDATE |
| `AFTER UPDATE` | Chạy sau khi UPDATE |
| `BEFORE DELETE` | Chạy trước khi DELETE |
| `AFTER DELETE` | Chạy sau khi DELETE |

> __Cẩn thận__: Trigger có thể gây __khó debug__ và __ảnh hưởng performance__. Nên dùng có chọn lọc.

## 8. Window Functions

### 8.1. Window Functions là gì?

__Window Functions__ thực hiện phép tính trên một __tập hàng liên quan__ (window) mà __không gộp__ các hàng lại (khác GROUP BY).

```sql
-- ROW_NUMBER: Đánh số thứ tự
SELECT name, department, salary,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rank_in_dept
FROM employees;

-- RANK: Xếp hạng (có thể trùng rank, bỏ qua)
SELECT name, salary,
    RANK() OVER (ORDER BY salary DESC) AS salary_rank
FROM employees;

-- DENSE_RANK: Xếp hạng (có thể trùng rank, không bỏ qua)
SELECT name, salary,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS salary_rank
FROM employees;

-- LAG / LEAD: Giá trị hàng trước / sau
SELECT name, salary,
    LAG(salary, 1) OVER (ORDER BY salary) AS prev_salary,
    LEAD(salary, 1) OVER (ORDER BY salary) AS next_salary
FROM employees;

-- SUM OVER: Tổng tích lũy
SELECT name, salary,
    SUM(salary) OVER (ORDER BY id) AS running_total
FROM employees;
```

<br />

### 8.2. ROW_NUMBER vs RANK vs DENSE_RANK

| Salary | ROW_NUMBER | RANK | DENSE_RANK |
| ------ | ---------- | ---- | ---------- |
| 20M | 1 | 1 | 1 |
| 18M | 2 | 2 | 2 |
| 18M | 3 | 2 | 2 |
| 15M | 4 | __4__ | __3__ |

- `ROW_NUMBER`: Luôn tuần tự (1, 2, 3, 4).
- `RANK`: Bỏ qua vị trí sau hàng trùng (1, 2, 2, __4__).
- `DENSE_RANK`: Không bỏ qua (1, 2, 2, __3__).

## 9. Câu hỏi phỏng vấn thường gặp

__Q: Index hoạt động thế nào? Tại sao không đánh index hết tất cả cột?__

Index tạo cấu trúc dữ liệu (thường B-Tree) giúp tra cứu nhanh. Không đánh hết vì: (1) Mỗi index tốn bộ nhớ lưu trữ, (2) Mỗi INSERT/UPDATE/DELETE phải cập nhật tất cả index → chậm ghi, (3) Cột ít giá trị phân biệt (gender) không hiệu quả.

__Q: View khác Table thế nào?__

Table lưu dữ liệu thật trên disk. View chỉ lưu câu query, mỗi lần truy cập sẽ chạy query. View giúp đơn giản hóa query phức tạp, bảo mật (ẩn cột nhạy cảm).

__Q: CTE khác Subquery thế nào?__

CTE được đặt tên, dễ đọc, có thể tái sử dụng trong cùng query, hỗ trợ recursive. Subquery inline, khó đọc khi lồng sâu. Performance thường tương đương.
