# Tối ưu hóa và Hệ thống phân tán

<br />

## 1. Query Optimization

### 1.1. Tại sao cần tối ưu Query?

Query chậm là nguyên nhân hàng đầu gây __giảm hiệu suất__ ứng dụng. Một query tối ưu có thể chạy __nhanh hơn hàng trăm lần__ so với query không tối ưu.

<br />

### 1.2. Nguyên tắc tối ưu

| Nguyên tắc | Mô tả |
| ---------- | ----- |
| __Chỉ SELECT cột cần thiết__ | Tránh `SELECT *`, chỉ lấy cột dùng |
| __Sử dụng Index hợp lý__ | Đánh index cho cột trong WHERE, JOIN, ORDER BY |
| __Tránh SELECT trong vòng lặp__ | Gộp thành 1 query hoặc dùng JOIN |
| __Dùng EXISTS thay vì IN__ | `EXISTS` dừng sớm khi tìm thấy, `IN` quét hết |
| __Giới hạn kết quả__ | Dùng `LIMIT` khi không cần tất cả |
| __Tránh function trên cột index__ | `WHERE YEAR(date) = 2024` → không dùng index |
| __Tránh LIKE bắt đầu bằng %__ | `LIKE '%keyword'` → Full Table Scan |

```sql
-- ❌ CHẬM: SELECT * và function trên cột index
SELECT * FROM orders
WHERE YEAR(created_at) = 2024 AND MONTH(created_at) = 6;

-- ✅ NHANH: Chỉ lấy cột cần, dùng range query
SELECT id, order_number, total_amount
FROM orders
WHERE created_at >= '2024-06-01' AND created_at < '2024-07-01';
```

```sql
-- ❌ CHẬM: IN với subquery lớn
SELECT * FROM employees
WHERE department_id IN (SELECT id FROM departments WHERE location = 'HCM');

-- ✅ NHANH: EXISTS hoặc JOIN
SELECT e.*
FROM employees e
WHERE EXISTS (
    SELECT 1 FROM departments d
    WHERE d.id = e.department_id AND d.location = 'HCM'
);

-- ✅ Hoặc dùng JOIN
SELECT e.*
FROM employees e
JOIN departments d ON e.department_id = d.id
WHERE d.location = 'HCM';
```

<br />

### 1.3. EXPLAIN — Phân tích Query

__EXPLAIN__ cho biết database sẽ thực thi query __như thế nào__ (execution plan).

```sql
EXPLAIN SELECT e.name, d.name
FROM employees e
JOIN departments d ON e.department_id = d.id
WHERE e.salary > 15000000;
```

Các thông tin quan trọng:

| Cột | Ý nghĩa | Giá trị tốt |
| --- | -------- | ------------ |
| __type__ | Kiểu truy cập | `const` > `ref` > `range` > `index` > `ALL` |
| __key__ | Index được sử dụng | Tên index (nếu NULL → không dùng index) |
| __rows__ | Số hàng ước tính phải quét | Càng nhỏ càng tốt |
| __Extra__ | Thông tin bổ sung | `Using index` (tốt), `Using filesort` (cần xem xét) |

| Type | Mô tả | Tốc độ |
| ---- | ----- | ------ |
| `const` / `eq_ref` | Truy cập bằng Primary Key / Unique Key | Nhanh nhất |
| `ref` | Truy cập bằng Non-Unique Index | Nhanh |
| `range` | Quét một khoảng index | Tốt |
| `index` | Quét toàn bộ index | Trung bình |
| `ALL` | __Full Table Scan__ — quét toàn bộ bảng | __Chậm nhất__ |

> __Quy tắc__: Nếu thấy `type = ALL` (Full Table Scan) trên bảng lớn → cần thêm index hoặc tối ưu query.

## 2. Partitioning — Phân vùng

### 2.1. Partitioning là gì?

__Partitioning__ chia một bảng lớn thành __nhiều phần nhỏ hơn__ (partition), mỗi phần được lưu trữ và quản lý riêng biệt nhưng vẫn được truy cập như một bảng duy nhất.

<br />

### 2.2. Các loại Partitioning

| Loại | Mô tả | Ví dụ |
| ---- | ----- | ----- |
| __Range Partitioning__ | Chia theo __khoảng giá trị__ | Theo năm: 2022, 2023, 2024 |
| __List Partitioning__ | Chia theo __danh sách giá trị cụ thể__ | Theo vùng: HN, HCM, DN |
| __Hash Partitioning__ | Chia theo __hàm hash__ | Hash(id) mod 4 → 4 partition |
| __Key Partitioning__ | Tương tự Hash, dùng __MySQL internal hash__ | — |
| __Composite Partitioning__ | Kết hợp nhiều loại | Range + Hash |

```sql
-- Range Partitioning: Chia orders theo năm
CREATE TABLE orders (
    id INT NOT NULL,
    order_date DATE NOT NULL,
    customer_id INT,
    total DECIMAL(10,2)
)
PARTITION BY RANGE (YEAR(order_date)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

```sql
-- List Partitioning: Chia theo khu vực
CREATE TABLE customers (
    id INT NOT NULL,
    name VARCHAR(100),
    region VARCHAR(20)
)
PARTITION BY LIST COLUMNS (region) (
    PARTITION p_north VALUES IN ('HN', 'HP', 'HY'),
    PARTITION p_central VALUES IN ('DN', 'HUE'),
    PARTITION p_south VALUES IN ('HCM', 'CT', 'BD')
);
```

> __Lợi ích__: Query chỉ quét partition liên quan (__partition pruning__) → nhanh hơn nhiều. Quản lý dữ liệu dễ hơn (xóa partition cũ thay vì DELETE).

<br />

### 2.3. Partitioning vs Sharding

| Tiêu chí | Partitioning | Sharding |
| -------- | ------------ | -------- |
| __Vị trí__ | Trong __cùng 1 database server__ | Trên __nhiều database server__ khác nhau |
| __Quản lý__ | Database tự quản lý | Application hoặc middleware quản lý |
| __Scale__ | Vertical (cùng server) | __Horizontal__ (nhiều server) |
| __Phức tạp__ | Đơn giản | Phức tạp (JOIN cross-shard, consistency) |

## 3. OLTP vs OLAP

### 3.1. So sánh

| Tiêu chí | OLTP | OLAP |
| -------- | ---- | ---- |
| __Tên đầy đủ__ | Online Transaction Processing | Online Analytical Processing |
| __Mục đích__ | Xử lý __giao dịch__ hàng ngày | __Phân tích__ dữ liệu lịch sử |
| __Thao tác__ | INSERT, UPDATE, DELETE nhiều | SELECT phức tạp, aggregate |
| __Dữ liệu__ | Hiện tại, chi tiết | Lịch sử, tổng hợp |
| __Thiết kế__ | Chuẩn hóa (3NF) | Denormalized (Star/Snowflake Schema) |
| __Response time__ | Milliseconds | Seconds đến minutes |
| __Ví dụ__ | Hệ thống đặt hàng, ngân hàng | Báo cáo doanh thu, dashboard BI |
| __Database__ | MySQL, PostgreSQL | BigQuery, Redshift, Snowflake |

<br />

### 3.2. Star Schema vs Snowflake Schema

| Tiêu chí | Star Schema | Snowflake Schema |
| -------- | ----------- | ---------------- |
| __Cấu trúc__ | 1 bảng Fact + nhiều bảng Dimension | Star Schema + Dimension được chuẩn hóa thêm |
| __JOIN__ | Ít JOIN (đơn giản) | Nhiều JOIN (phức tạp) |
| __Dư thừa__ | Có thể có | Ít dư thừa |
| __Performance__ | Nhanh hơn (ít JOIN) | Chậm hơn |
| __Dung lượng__ | Lớn hơn | Nhỏ hơn |

## 4. ETL Process

### 4.1. ETL là gì?

__ETL (Extract, Transform, Load)__ là quy trình chuyển dữ liệu từ __nhiều nguồn__ vào __Data Warehouse__ để phân tích.

| Bước | Mô tả | Ví dụ |
| ---- | ----- | ----- |
| __Extract__ | Trích xuất dữ liệu từ nhiều nguồn | MySQL, CSV, API, logs |
| __Transform__ | Làm sạch, chuyển đổi, chuẩn hóa dữ liệu | Loại bỏ trùng, đổi format ngày, tính toán |
| __Load__ | Nạp dữ liệu đã xử lý vào Data Warehouse | Load vào BigQuery, Redshift |

```
[MySQL] ──┐
[CSV]   ──┼── Extract → Transform → Load → [Data Warehouse] → [BI Dashboard]
[API]   ──┘
```

> __ETL vs ELT__: ETL transform __trước__ khi load. ELT load raw data trước, transform __trong__ Data Warehouse. ELT phù hợp với cloud DW hiện đại (BigQuery, Snowflake).

## 5. Replication

### 5.1. Database Replication là gì?

__Replication__ (Nhân bản) là quá trình sao chép dữ liệu từ một database (__master/primary__) sang một hoặc nhiều database khác (__slave/replica__).

<br />

### 5.2. Các mô hình Replication

| Mô hình | Mô tả | Ưu điểm | Nhược điểm |
| ------- | ----- | -------- | ---------- |
| __Master-Slave__ | 1 Master (ghi) → N Slaves (đọc) | Đơn giản, tăng throughput đọc | Master là single point of failure |
| __Master-Master__ | Nhiều Master đều ghi được | Không SPOF, write scalability | Xung đột dữ liệu |
| __Synchronous__ | Slave xác nhận nhận được → Master mới commit | Dữ liệu nhất quán | Chậm (chờ Slave) |
| __Asynchronous__ | Master commit ngay, Slave nhận sau | Nhanh | Có thể mất dữ liệu nếu Master down |

```
         ┌── Slave 1 (READ)
Master ──┤── Slave 2 (READ)
(WRITE)  └── Slave 3 (READ)
```

> __Use case phổ biến__: Master xử lý __write__, Slaves xử lý __read__. Ứng dụng kết nối Master để ghi, kết nối Slave (qua load balancer) để đọc → __phân tải hiệu quả__.

## 6. CAP Theorem

### 6.1. CAP Theorem là gì?

__CAP Theorem__ (Định lý CAP) phát biểu rằng một hệ thống phân tán __chỉ có thể đảm bảo 2 trong 3__ tính chất:

| Tính chất | Mô tả |
| --------- | ----- |
| __Consistency__ (C) | Mọi node đều trả về __cùng dữ liệu__ tại cùng thời điểm |
| __Availability__ (A) | Hệ thống __luôn phản hồi__ (dù có thể không phải dữ liệu mới nhất) |
| __Partition Tolerance__ (P) | Hệ thống __vẫn hoạt động__ khi mạng bị phân mảnh |

| Hệ thống | Chọn | Bỏ | Ví dụ |
| --------- | ---- | --- | ----- |
| __CP__ | Consistency + Partition Tolerance | Availability | MongoDB, HBase, Redis |
| __AP__ | Availability + Partition Tolerance | Consistency | Cassandra, DynamoDB, CouchDB |
| __CA__ | Consistency + Availability | Partition Tolerance | Traditional RDBMS (single node) |

> Trong thực tế, __Partition Tolerance__ là bắt buộc cho hệ phân tán → lựa chọn thường là __CP__ hoặc __AP__.

## 7. Connection Pooling

### 7.1. Connection Pooling là gì?

__Connection Pooling__ (Bể kết nối) là kỹ thuật duy trì một __nhóm kết nối database sẵn sàng__ để tái sử dụng, thay vì tạo/đóng kết nối mới cho mỗi request.

```
Application ──→ Connection Pool ──→ Database
                   ┌── Connection 1 (busy)
                   ├── Connection 2 (idle) → reuse
                   ├── Connection 3 (busy)
                   └── Connection 4 (idle) → reuse
```

__Lợi ích:__
- __Giảm overhead__ tạo/đóng kết nối (tốn nhiều tài nguyên).
- __Giới hạn số kết nối__ đồng thời (tránh quá tải database).
- __Tăng throughput__ ứng dụng.

```properties
# Spring Boot: HikariCP (default connection pool)
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.idle-timeout=300000
spring.datasource.hikari.connection-timeout=20000
spring.datasource.hikari.max-lifetime=1200000
```

| Tham số | Mô tả | Giá trị gợi ý |
| ------- | ----- | -------------- |
| `maximum-pool-size` | Số kết nối tối đa | 10-20 (tùy tải) |
| `minimum-idle` | Số kết nối idle tối thiểu | 5 |
| `idle-timeout` | Thời gian idle tối đa trước khi đóng | 300000 (5 phút) |
| `connection-timeout` | Thời gian chờ lấy kết nối từ pool | 20000 (20 giây) |

## 8. Cursor

### 8.1. Cursor là gì?

__Cursor__ là một __con trỏ__ cho phép duyệt qua __từng hàng__ của result set. Cursor hữu ích khi cần xử lý logic phức tạp trên từng hàng.

```sql
DELIMITER //
CREATE PROCEDURE UpdateSalaries()
BEGIN
    DECLARE done INT DEFAULT FALSE;
    DECLARE emp_id INT;
    DECLARE emp_salary DECIMAL(10,2);

    -- Khai báo cursor
    DECLARE emp_cursor CURSOR FOR
        SELECT id, salary FROM employees WHERE status = 'ACTIVE';

    -- Handler khi hết dữ liệu
    DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

    OPEN emp_cursor;

    read_loop: LOOP
        FETCH emp_cursor INTO emp_id, emp_salary;
        IF done THEN
            LEAVE read_loop;
        END IF;

        -- Xử lý từng hàng
        IF emp_salary < 10000000 THEN
            UPDATE employees SET salary = salary * 1.10 WHERE id = emp_id;
        END IF;
    END LOOP;

    CLOSE emp_cursor;
END //
DELIMITER ;
```

> __Cẩn thận__: Cursor xử lý __từng hàng__ → __chậm__ với dữ liệu lớn. Nên ưu tiên dùng __set-based operations__ (UPDATE, JOIN) thay vì cursor khi có thể.

## 9. Câu hỏi phỏng vấn thường gặp

__Q: Làm thế nào để tối ưu query chậm?__

(1) Dùng EXPLAIN để phân tích execution plan. (2) Thêm index cho cột trong WHERE/JOIN/ORDER BY. (3) Tránh SELECT *, chỉ lấy cột cần. (4) Tránh function trên cột index. (5) Dùng LIMIT. (6) Cân nhắc denormalization nếu cần.

__Q: Partitioning khác Sharding thế nào?__

Partitioning chia bảng thành phần nhỏ trong cùng 1 server. Sharding chia dữ liệu trên nhiều server khác nhau. Partitioning đơn giản hơn, Sharding scale tốt hơn nhưng phức tạp hơn nhiều.

__Q: CAP Theorem là gì? Giải thích.__

Hệ thống phân tán chỉ đảm bảo 2/3: Consistency (dữ liệu nhất quán), Availability (luôn phản hồi), Partition Tolerance (chịu được mạng bị chia cắt). Thực tế P là bắt buộc, nên chọn CP (nhất quán) hoặc AP (khả dụng).

__Q: OLTP khác OLAP thế nào?__

OLTP xử lý giao dịch hàng ngày (insert/update, dữ liệu hiện tại, normalized, response nhanh). OLAP phân tích dữ liệu (aggregate, dữ liệu lịch sử, denormalized, query phức tạp).
