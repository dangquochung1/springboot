# Transaction và Bảo mật Database

<br />

## 1. Transaction

### 1.1. Transaction là gì?

__Transaction__ (Giao dịch) là một nhóm các thao tác SQL được thực hiện như __một đơn vị công việc duy nhất__. Hoặc tất cả thành công (__COMMIT__), hoặc tất cả thất bại (__ROLLBACK__).

```sql
-- Ví dụ: Chuyển tiền từ tài khoản A sang B
START TRANSACTION;

UPDATE accounts SET balance = balance - 5000000 WHERE id = 1;  -- Trừ A
UPDATE accounts SET balance = balance + 5000000 WHERE id = 2;  -- Cộng B

-- Nếu không có lỗi
COMMIT;

-- Nếu có lỗi
ROLLBACK;
```

<br />

### 1.2. ACID Properties

Mọi transaction phải đảm bảo 4 tính chất __ACID__:

| Tính chất | Mô tả | Ví dụ |
| --------- | ----- | ----- |
| __Atomicity__ (Nguyên tử) | Tất cả hoặc không gì cả | Chuyển tiền: trừ A + cộng B cùng thành công/rollback |
| __Consistency__ (Nhất quán) | Database luôn ở trạng thái hợp lệ trước và sau transaction | Tổng tiền trong hệ thống không đổi |
| __Isolation__ (Cô lập) | Các transaction đồng thời không ảnh hưởng lẫn nhau | 2 người cùng rút tiền, mỗi người thấy số dư đúng |
| __Durability__ (Bền vững) | Dữ liệu đã COMMIT không mất khi hệ thống gặp sự cố | Mất điện sau COMMIT → dữ liệu vẫn còn |

<br />

### 1.3. SAVEPOINT

__SAVEPOINT__ cho phép đặt điểm lưu trong transaction để __rollback một phần__ thay vì toàn bộ.

```sql
START TRANSACTION;

INSERT INTO orders (customer_id, total) VALUES (1, 500000);
SAVEPOINT sp_order_created;

INSERT INTO order_items (order_id, product_id, qty) VALUES (1, 101, 2);
INSERT INTO order_items (order_id, product_id, qty) VALUES (1, 102, 1);

-- Giả sử item thứ 2 bị lỗi, chỉ rollback đến savepoint
ROLLBACK TO sp_order_created;
-- Order vẫn còn, chỉ order_items bị rollback

COMMIT;
```

## 2. Isolation Levels

### 2.1. Các vấn đề concurrency

Khi nhiều transaction chạy đồng thời, có thể phát sinh các vấn đề:

| Vấn đề | Mô tả |
| ------ | ----- |
| __Dirty Read__ | Transaction đọc dữ liệu __chưa commit__ của transaction khác |
| __Non-Repeatable Read__ | Cùng 1 query trong 1 transaction cho __kết quả giá trị khác__ (do transaction khác UPDATE + COMMIT) |
| __Phantom Read__ | Cùng 1 query cho ra __số lượng hàng khác__ (do transaction khác INSERT/DELETE + COMMIT) |
| __Lost Update__ | 2 transaction cùng update 1 row, transaction sau ghi đè transaction trước |

<br />

### 2.2. Các Isolation Level

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Performance |
| ----- | ---------- | ------------------- | ------------ | ----------- |
| __READ UNCOMMITTED__ | ✅ Có thể | ✅ Có thể | ✅ Có thể | Nhanh nhất |
| __READ COMMITTED__ | ❌ Ngăn | ✅ Có thể | ✅ Có thể | Nhanh |
| __REPEATABLE READ__ | ❌ Ngăn | ❌ Ngăn | ✅ Có thể | Trung bình |
| __SERIALIZABLE__ | ❌ Ngăn | ❌ Ngăn | ❌ Ngăn | Chậm nhất |

```sql
-- Đặt Isolation Level
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
START TRANSACTION;
-- ...
COMMIT;
```

> __Mặc định__: MySQL dùng `REPEATABLE READ`, PostgreSQL và SQL Server dùng `READ COMMITTED`.

## 3. Concurrency Control

### 3.1. Lock — Khóa

__Lock__ (Khóa) là cơ chế ngăn nhiều transaction truy cập đồng thời cùng dữ liệu.

| Loại Lock | Mô tả | Ví dụ |
| --------- | ----- | ----- |
| __Shared Lock (S)__ | Nhiều transaction có thể __đọc__ cùng lúc | `SELECT ... LOCK IN SHARE MODE` |
| __Exclusive Lock (X)__ | Chỉ 1 transaction được __đọc và ghi__ | `SELECT ... FOR UPDATE` |
| __Row-level Lock__ | Khóa từng hàng | InnoDB mặc định |
| __Table-level Lock__ | Khóa toàn bộ bảng | MyISAM, hoặc `LOCK TABLES` |

```sql
-- Exclusive Lock: Đọc và khóa hàng để update
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Hàng bị khóa, transaction khác phải chờ
UPDATE accounts SET balance = balance - 1000000 WHERE id = 1;
COMMIT;
```

<br />

### 3.2. Deadlock

__Deadlock__ xảy ra khi 2 transaction chờ lẫn nhau giải phóng lock → cả 2 bị kẹt vĩnh viễn.

```
Transaction A: Lock row 1 → chờ lock row 2
Transaction B: Lock row 2 → chờ lock row 1
→ DEADLOCK!
```

__Phòng tránh Deadlock:__
- Truy cập tài nguyên theo __cùng thứ tự__.
- Giữ transaction __ngắn nhất có thể__.
- Sử dụng __timeout__.
- Database tự động phát hiện và __rollback 1 transaction__ (victim).

<br />

### 3.3. Optimistic vs Pessimistic Locking

| Tiêu chí | Optimistic Locking | Pessimistic Locking |
| -------- | ------------------ | ------------------- |
| __Giả định__ | Xung đột ít xảy ra | Xung đột thường xuyên |
| __Cơ chế__ | Kiểm tra version khi commit | Khóa dữ liệu ngay khi đọc |
| __Lock__ | Không lock khi đọc | Lock khi đọc |
| __Phát hiện xung đột__ | Khi UPDATE (check version) | Ngăn xung đột từ đầu |
| __Performance__ | Tốt khi ít xung đột | Tốt khi nhiều xung đột |
| __Cách triển khai__ | Cột `version` hoặc `updated_at` | `SELECT ... FOR UPDATE` |

```sql
-- Optimistic Locking: Dùng cột version
-- Bước 1: Đọc (không lock)
SELECT id, balance, version FROM accounts WHERE id = 1;
-- version = 5

-- Bước 2: Update với kiểm tra version
UPDATE accounts
SET balance = balance - 1000000, version = version + 1
WHERE id = 1 AND version = 5;
-- Nếu 0 rows affected → có người khác đã update → retry

-- Pessimistic Locking: Lock ngay
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Row bị lock, transaction khác phải chờ
UPDATE accounts SET balance = balance - 1000000 WHERE id = 1;
COMMIT;
```

## 4. Bảo mật Database

### 4.1. Authentication và Authorization

| Khái niệm | Mô tả |
| ---------- | ----- |
| __Authentication__ (Xác thực) | Xác minh danh tính người dùng (username/password) |
| __Authorization__ (Phân quyền) | Xác định quyền hạn (đọc, ghi, admin) |

```sql
-- Tạo user
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'strong_password';

-- Cấp quyền cụ thể
GRANT SELECT, INSERT, UPDATE ON mydb.employees TO 'app_user'@'localhost';

-- Cấp quyền admin
GRANT ALL PRIVILEGES ON mydb.* TO 'admin_user'@'localhost';

-- Thu hồi quyền
REVOKE DELETE ON mydb.employees FROM 'app_user'@'localhost';

-- Xem quyền
SHOW GRANTS FOR 'app_user'@'localhost';
```

<br />

### 4.2. SQL Injection

__SQL Injection__ là kỹ thuật tấn công chèn mã SQL độc hại vào input để thao túng database.

```sql
-- ❌ Code dễ bị SQL Injection
String query = "SELECT * FROM users WHERE email = '" + email + "' AND password = '" + password + "'";
-- Nếu email = "' OR 1=1 --"
-- → SELECT * FROM users WHERE email = '' OR 1=1 --' AND password = ''
-- → Trả về TẤT CẢ users!
```

__Phòng chống SQL Injection:__

| Phương pháp | Mô tả |
| ----------- | ----- |
| __Prepared Statements__ | Tham số hóa query, tách biệt code và dữ liệu |
| __Stored Procedures__ | Logic nằm trong DB, client chỉ gọi |
| __Input Validation__ | Kiểm tra, lọc input đầu vào |
| __ORM__ | Sử dụng Hibernate, JPA (tự động parameterize) |
| __Least Privilege__ | Cấp quyền tối thiểu cho DB user |

```java
// ✅ ĐÚNG: Prepared Statement (Java)
String sql = "SELECT * FROM users WHERE email = ? AND password = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setString(1, email);
pstmt.setString(2, password);
ResultSet rs = pstmt.executeQuery();
```

### 4.3. Mã hóa dữ liệu (Encryption)

| Loại | Mô tả | Ví dụ |
| ---- | ----- | ----- |
| __At Rest__ | Mã hóa dữ liệu __trên disk__ | TDE (Transparent Data Encryption) |
| __In Transit__ | Mã hóa dữ liệu __truyền qua mạng__ | SSL/TLS connection |
| __Column-level__ | Mã hóa __cột cụ thể__ chứa dữ liệu nhạy cảm | Mã hóa cột password, SSN |

```sql
-- MySQL: Mã hóa cột
INSERT INTO users (name, email, ssn)
VALUES ('An', 'an@example.com', AES_ENCRYPT('123456789', 'secret_key'));

-- Giải mã
SELECT name, email, AES_DECRYPT(ssn, 'secret_key') AS ssn FROM users;
```

## 5. Backup và Recovery

### 5.1. Các chiến lược Backup

| Loại | Mô tả | Thời gian | Dung lượng |
| ---- | ----- | --------- | ---------- |
| __Full Backup__ | Sao lưu __toàn bộ__ database | Lâu nhất | Lớn nhất |
| __Incremental Backup__ | Chỉ sao lưu __thay đổi__ kể từ backup trước | Nhanh | Nhỏ |
| __Differential Backup__ | Sao lưu thay đổi kể từ __full backup gần nhất__ | Trung bình | Trung bình |
| __Point-in-Time Recovery__ | Khôi phục đến __thời điểm bất kỳ__ | — | Cần binary log |

```bash
# MySQL: Full Backup
mysqldump -u root -p --all-databases > full_backup.sql

# MySQL: Restore
mysql -u root -p < full_backup.sql

# PostgreSQL: Backup
pg_dump mydb > backup.sql

# PostgreSQL: Restore
psql mydb < backup.sql
```

> __Best Practice__:
> - Full backup __hàng tuần__.
> - Incremental/Differential backup __hàng ngày__.
> - Test restore __định kỳ__ để đảm bảo backup hoạt động.
> - Lưu backup ở __nơi khác__ (offsite, cloud).

## 6. Câu hỏi phỏng vấn thường gặp

__Q: ACID là gì? Cho ví dụ.__

ACID gồm: Atomicity (nguyên tử — all or nothing), Consistency (nhất quán — DB hợp lệ trước/sau), Isolation (cô lập — các transaction không ảnh hưởng nhau), Durability (bền vững — dữ liệu commit không mất). Ví dụ: chuyển tiền ngân hàng.

__Q: Deadlock là gì? Cách phòng tránh?__

Deadlock xảy ra khi 2+ transactions chờ lẫn nhau giải phóng lock. Phòng tránh: truy cập tài nguyên cùng thứ tự, giữ transaction ngắn, dùng timeout, DB tự detect và kill 1 transaction.

__Q: Optimistic vs Pessimistic Locking?__

Optimistic: không lock khi đọc, check version khi update, phù hợp ít xung đột. Pessimistic: lock ngay khi đọc (SELECT FOR UPDATE), phù hợp nhiều xung đột. Web apps thường dùng Optimistic.

__Q: SQL Injection là gì? Cách phòng chống?__

SQL Injection chèn mã SQL qua input. Phòng chống: dùng Prepared Statements / Parameterized Queries, không concat string vào SQL, validate input, dùng ORM, cấp quyền tối thiểu cho DB user.
