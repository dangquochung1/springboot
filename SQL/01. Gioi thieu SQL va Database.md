# Giới thiệu SQL và Database

<br />

## 1. SQL là gì?

### 1.1. Khái niệm

__SQL (Structured Query Language)__ là ngôn ngữ truy vấn được thiết kế đặc biệt để làm việc với các __hệ quản trị cơ sở dữ liệu quan hệ (RDBMS)__. SQL được sử dụng để lưu trữ, truy xuất, quản lý và thao tác dữ liệu.

SQL hỗ trợ:
- __Truy vấn dữ liệu__ (query): Tìm kiếm, lọc, sắp xếp dữ liệu.
- __Thao tác dữ liệu__ (manipulation): Thêm, sửa, xóa dữ liệu.
- __Quản trị database__: Tạo bảng, cấp quyền, định nghĩa cấu trúc.

SQL được phát triển lần đầu vào những năm 1970 tại __IBM__, sau đó được chuẩn hóa bởi __ANSI__ và __ISO__, trở thành ngôn ngữ chuẩn quốc tế cho cơ sở dữ liệu quan hệ.

<br />

### 1.2. Các nhóm câu lệnh SQL

SQL được chia thành 3 nhóm ngôn ngữ con chính:

| Nhóm | Tên đầy đủ | Mục đích | Câu lệnh |
| ---- | ---------- | -------- | --------- |
| __DDL__ | Data Definition Language | Định nghĩa cấu trúc database | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` |
| __DML__ | Data Manipulation Language | Thao tác với dữ liệu | `SELECT`, `INSERT`, `UPDATE`, `DELETE` |
| __DCL__ | Data Control Language | Quản lý quyền truy cập | `GRANT`, `REVOKE` |

Ví dụ:
```sql
-- DDL: Tạo bảng
CREATE TABLE employees (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(150) UNIQUE,
    salary DECIMAL(10, 2),
    department VARCHAR(50)
);

-- DML: Thêm dữ liệu
INSERT INTO employees (name, email, salary, department)
VALUES ('Nguyen Van A', 'a@example.com', 15000000, 'Engineering');

-- DCL: Cấp quyền
GRANT SELECT ON employees TO 'read_user'@'localhost';
```

## 2. Các loại Database

### 2.1. Tổng quan

| Loại | Mô tả | Ví dụ |
| ---- | ----- | ----- |
| __Relational Database__ | Dữ liệu lưu trữ dưới dạng bảng có hàng và cột, quan hệ qua khóa | MySQL, PostgreSQL, SQL Server, Oracle |
| __NoSQL Database__ | Không theo mô hình quan hệ, linh hoạt về schema | MongoDB, Cassandra, Redis, CouchDB |
| __Hierarchical Database__ | Dữ liệu tổ chức dạng cây (cha - con) | IBM IMS |
| __Distributed Database__ | Dữ liệu phân tán trên nhiều máy chủ/vị trí địa lý | CockroachDB, Google Spanner |
| __Object-Oriented Database__ | Lưu trữ dữ liệu dưới dạng đối tượng (objects) | db4o, ObjectDB |
| __Cloud Database__ | Được quản lý và vận hành trên nền tảng cloud | Amazon RDS, Google Cloud SQL, Azure SQL |
| __In-Memory Database__ | Lưu trữ dữ liệu chính trong RAM để truy xuất cực nhanh | Redis, Memcached, SAP HANA |

<br />

### 2.2. SQL vs NoSQL

| Tiêu chí | SQL | NoSQL |
| -------- | --- | ----- |
| __Loại CSDL__ | Quan hệ (Relational) | Linh hoạt (Key-Value, Document, Graph,...) |
| __Mô hình dữ liệu__ | Bảng có hàng và cột | Key-Value, Document, Column-Family, Graph |
| __Cấu trúc__ | Schema cố định, chặt chẽ | Schema linh hoạt hoặc động |
| __Ngôn ngữ truy vấn__ | SQL chuẩn | API hoặc ngôn ngữ riêng |
| __Khả năng mở rộng__ | Vertical Scaling (chủ yếu) | Horizontal Scaling |
| __Tính nhất quán__ | Tuân thủ __ACID__ | Tuân thủ __BASE__ (Basically Available, Soft state, Eventually consistent) |
| __Phù hợp cho__ | Dữ liệu có cấu trúc rõ ràng, cần tính toàn vẹn | Dữ liệu linh hoạt, cần scale lớn |
| __Ví dụ__ | MySQL, PostgreSQL, Oracle | MongoDB, Cassandra, Redis |

> __Khi nào dùng SQL?__ Dữ liệu có cấu trúc rõ ràng, cần đảm bảo ACID (tài chính, ERP). __Khi nào dùng NoSQL?__ Dữ liệu không cấu trúc, cần scale nhanh (social media, IoT, real-time analytics).

## 3. Schema

### 3.1. Database Schema là gì?

__Schema__ là bản thiết kế (blueprint) mô tả cấu trúc logic của database: các bảng, cột, kiểu dữ liệu, quan hệ, ràng buộc.

```sql
-- Ví dụ Schema đơn giản
CREATE TABLE departments (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL UNIQUE
);

CREATE TABLE employees (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(150) UNIQUE,
    salary DECIMAL(10, 2),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(id)
);
```

<br />

### 3.2. Schema-based vs Schema-less

| Tiêu chí | Schema-based (SQL) | Schema-less (NoSQL) |
| -------- | ------------------ | ------------------- |
| __Cấu trúc__ | Phải định nghĩa trước khi chèn dữ liệu | Không cần định nghĩa trước |
| __Thay đổi__ | Cần `ALTER TABLE` để thay đổi cấu trúc | Thêm field mới bất kỳ lúc nào |
| __Ưu điểm__ | Đảm bảo tính toàn vẹn, nhất quán | Linh hoạt, nhanh chóng thích ứng |
| __Nhược điểm__ | Khó thay đổi khi đã có dữ liệu lớn | Khó kiểm soát tính nhất quán |

## 4. Database Engine

### 4.1. Các loại Database Engine

__Database Engine__ (Storage Engine) là thành phần cốt lõi xử lý việc lưu trữ, truy xuất và quản lý dữ liệu.

| Loại | Engine | Đặc điểm |
| ---- | ------ | -------- |
| __Storage Engine__ | InnoDB (MySQL) | Hỗ trợ ACID, transaction, row-level locking |
| | MyISAM (MySQL) | Nhanh cho đọc, không hỗ trợ transaction |
| | PostgreSQL Engine | MVCC, hỗ trợ đầy đủ ACID |
| | SQL Server | Tích hợp BI, Analysis Services |
| __Search Engine__ | Elasticsearch | Full-text search, phân tán |
| | Sphinx | Tìm kiếm văn bản nhanh |
| __In-Memory Engine__ | Redis | Key-Value store, cache, pub/sub |
| | Memcached | Distributed caching |
| | SAP HANA | In-memory computing, analytics |

## 5. Primary Key và Foreign Key

### 5.1. Phân biệt

| Tiêu chí | Primary Key | Foreign Key |
| -------- | ----------- | ----------- |
| __Định nghĩa__ | Khóa chính, xác định tính duy nhất mỗi hàng | Khóa ngoại, tạo mối quan hệ giữa các bảng |
| __Tính duy nhất__ | Không trùng lặp, không NULL | Có thể trùng lặp, có thể NULL |
| __Bảng áp dụng__ | Bảng chính | Bảng con (tham chiếu đến PK bảng chính) |
| __Số lượng__ | Mỗi bảng chỉ có __1__ Primary Key | Một bảng có thể có __nhiều__ Foreign Key |
| __Mục đích__ | Đảm bảo tính duy nhất và toàn vẹn | Tạo ràng buộc toàn vẹn giữa 2 bảng |

```sql
-- Primary Key
CREATE TABLE departments (
    id INT PRIMARY KEY AUTO_INCREMENT,  -- PK
    name VARCHAR(100) NOT NULL
);

-- Foreign Key
CREATE TABLE employees (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(id)  -- FK
);
```

<br />

### 5.2. Composite Key

__Composite Key__ là khóa chính được tạo từ __tổ hợp nhiều cột__. Mỗi cột riêng lẻ có thể trùng, nhưng tổ hợp phải là duy nhất.

```sql
CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    enrolled_at DATE,
    PRIMARY KEY (student_id, course_id),  -- Composite Key
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id) REFERENCES courses(id)
);
```

## 6. Thiết kế Database

### 6.1. Logical vs Physical Design

| Tiêu chí | Logical Design | Physical Design |
| -------- | -------------- | --------------- |
| __Mức độ__ | Mức khái niệm, trừu tượng | Mức triển khai, cụ thể |
| __Nội dung__ | Entity, Relationship, Attribute | Table, Column, Index, Datatype |
| __Công cụ__ | ER Diagram (Entity-Relationship) | SQL DDL Statements |
| __Phụ thuộc DBMS__ | Không | Có (MySQL, PostgreSQL,...) |

<br />

### 6.2. Data Warehouse

__Data Warehouse__ (Kho dữ liệu) là hệ thống lưu trữ dữ liệu tổng hợp từ nhiều nguồn, phục vụ cho phân tích và báo cáo.

| Tiêu chí | Traditional Database | Data Warehouse |
| -------- | -------------------- | -------------- |
| __Mục đích__ | Xử lý giao dịch hàng ngày (OLTP) | Phân tích dữ liệu lịch sử (OLAP) |
| __Dữ liệu__ | Dữ liệu hiện tại, chi tiết | Dữ liệu lịch sử, tổng hợp |
| __Thiết kế__ | Chuẩn hóa (3NF) | Star Schema / Snowflake Schema |
| __Truy vấn__ | Đơn giản, nhanh | Phức tạp, quét nhiều dữ liệu |

## 7. Câu hỏi phỏng vấn thường gặp

__Q: SQL là gì? Kể tên các loại câu lệnh SQL.__

SQL là ngôn ngữ truy vấn cấu trúc dùng cho RDBMS. Ba nhóm chính: DDL (CREATE, ALTER, DROP), DML (SELECT, INSERT, UPDATE, DELETE), DCL (GRANT, REVOKE).

__Q: SQL khác NoSQL như thế nào?__

SQL dùng mô hình quan hệ (bảng), schema cố định, tuân thủ ACID, scale vertical. NoSQL dùng mô hình linh hoạt (document, key-value,...), schema động, tuân thủ BASE, scale horizontal.

__Q: Primary Key khác Foreign Key như thế nào?__

PK xác định tính duy nhất mỗi hàng (không NULL, không trùng, chỉ 1 per table). FK tạo quan hệ giữa 2 bảng, tham chiếu đến PK bảng khác (có thể NULL, có thể trùng, nhiều FK per table).
