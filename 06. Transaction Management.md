# Transaction Management

<br />

## 1. Transaction là gì?

### 1.1. Khái niệm

__Transaction__ (giao dịch) là một nhóm các thao tác với database được thực hiện như __một đơn vị công việc duy nhất__. Tất cả các thao tác trong transaction phải thành công hoàn toàn hoặc thất bại hoàn toàn — không có trạng thái trung gian.

Ví dụ thực tế: Chuyển tiền từ tài khoản A sang tài khoản B:
1. Trừ tiền tài khoản A.
2. Cộng tiền tài khoản B.

→ Cả 2 thao tác phải thành công hoặc cùng thất bại. Không thể chỉ trừ tiền A mà không cộng tiền B.

<br />

### 1.2. ACID Properties

Mọi transaction đều phải đảm bảo 4 tính chất __ACID__:

| Tính chất | Ý nghĩa | Ví dụ |
| --------- | -------- | ----- |
| __Atomicity__ (Nguyên tử) | Tất cả hoặc không gì cả | Chuyển tiền: trừ A + cộng B cùng thành công hoặc cùng rollback |
| __Consistency__ (Nhất quán) | Database luôn ở trạng thái hợp lệ | Tổng tiền trước và sau chuyển khoản phải bằng nhau |
| __Isolation__ (Cô lập) | Các transaction đồng thời không ảnh hưởng lẫn nhau | 2 người cùng rút tiền, mỗi người thấy số dư chính xác |
| __Durability__ (Bền vững) | Dữ liệu đã commit không bị mất | Mất điện sau commit → dữ liệu vẫn được lưu |

## 2. `@Transactional` trong Spring

### 2.1. Cách sử dụng cơ bản

Spring cung cấp annotation `@Transactional` để quản lý transaction một cách __declarative__ (khai báo), không cần viết code quản lý transaction thủ công.

```java
@Service
@RequiredArgsConstructor
public class TransferService {

    private final AccountRepository accountRepository;

    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId)
                .orElseThrow(() -> new RuntimeException("Account not found"));
        Account to = accountRepository.findById(toId)
                .orElseThrow(() -> new RuntimeException("Account not found"));

        if (from.getBalance().compareTo(amount) < 0) {
            throw new InsufficientBalanceException("Insufficient balance");
        }

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
        // Nếu exception xảy ra → tự động ROLLBACK cả 2 thao tác
        // Nếu thành công → tự động COMMIT
    }
}
```

<br />

### 2.2. Vị trí đặt `@Transactional`

| Vị trí | Phạm vi |
| ------ | ------- |
| Trên __method__ | Chỉ method đó có transaction |
| Trên __class__ | Tất cả public methods trong class đều có transaction |
| Trên __interface__ | Không khuyến khích (có thể không hoạt động với proxy) |

```java
@Service
@Transactional  // Áp dụng cho tất cả public methods
public class UserService {

    public User findById(Long id) { ... }         // Có transaction

    @Transactional(readOnly = true)                // Override: read-only transaction
    public List<User> findAll() { ... }

    public User save(User user) { ... }            // Có transaction (kế thừa từ class)
}
```

> __Best Practice__: Đặt `@Transactional(readOnly = true)` trên class (mặc định read-only), rồi override `@Transactional` trên các method ghi (create, update, delete).

```java
@Service
@Transactional(readOnly = true)  // Mặc định read-only
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;

    public User findById(Long id) { ... }    // read-only ✅
    public List<User> findAll() { ... }       // read-only ✅

    @Transactional                            // Override: read-write
    public User create(User user) { ... }

    @Transactional
    public void delete(Long id) { ... }
}
```

## 3. Transaction Propagation

### 3.1. Propagation là gì?

__Propagation__ xác định cách transaction __mới__ liên kết với transaction __hiện tại__ (nếu có).

```java
@Transactional(propagation = Propagation.REQUIRED)
public void methodA() { ... }
```

<br />

### 3.2. Các loại Propagation

| Propagation | Mô tả |
| ----------- | ----- |
| __REQUIRED__ (mặc định) | Dùng transaction hiện tại, nếu chưa có thì tạo mới |
| __REQUIRES_NEW__ | Luôn tạo transaction mới, tạm dừng transaction hiện tại |
| __NESTED__ | Tạo savepoint trong transaction hiện tại |
| __SUPPORTS__ | Dùng transaction hiện tại nếu có, không có thì chạy không transaction |
| __NOT_SUPPORTED__ | Luôn chạy không transaction, tạm dừng transaction hiện tại |
| __MANDATORY__ | Bắt buộc phải có transaction hiện tại, nếu không → exception |
| __NEVER__ | Bắt buộc không có transaction, nếu có → exception |

<br />

### 3.3. Ví dụ: REQUIRED vs REQUIRES_NEW

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final AuditLogService auditLogService;

    @Transactional  // REQUIRED (mặc định)
    public void createOrder(OrderRequest request) {
        Order order = new Order(request);
        orderRepository.save(order);

        // auditLogService.log() sẽ CHUNG transaction với createOrder()
        // Nếu log() fail → createOrder() cũng ROLLBACK
        auditLogService.log("Order created: " + order.getId());
    }
}

@Service
@RequiredArgsConstructor
public class AuditLogService {

    private final AuditLogRepository auditLogRepository;

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String message) {
        // Luôn tạo TRANSACTION MỚI, độc lập với transaction gọi nó
        // Nếu log() fail → KHÔNG ảnh hưởng createOrder()
        auditLogRepository.save(new AuditLog(message));
    }
}
```

## 4. Isolation Levels

### 4.1. Isolation Level là gì?

__Isolation Level__ xác định mức độ __cô lập__ giữa các transaction đồng thời, ảnh hưởng đến việc một transaction có thể __nhìn thấy__ dữ liệu thay đổi bởi transaction khác hay không.

<br />

### 4.2. Các vấn đề khi thiếu cô lập

| Vấn đề | Mô tả |
| ------ | ----- |
| __Dirty Read__ | Đọc được dữ liệu __chưa commit__ của transaction khác |
| __Non-Repeatable Read__ | Cùng 1 query trong 1 transaction cho kết quả khác nhau (do transaction khác update + commit) |
| __Phantom Read__ | Cùng 1 query cho ra __số lượng row khác nhau__ (do transaction khác insert/delete + commit) |

<br />

### 4.3. Các Isolation Level

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Performance |
| ----- | ---------- | ------------------- | ------------ | ----------- |
| __READ_UNCOMMITTED__ | ✅ Có thể | ✅ Có thể | ✅ Có thể | Nhanh nhất |
| __READ_COMMITTED__ | ❌ Ngăn | ✅ Có thể | ✅ Có thể | Nhanh |
| __REPEATABLE_READ__ | ❌ Ngăn | ❌ Ngăn | ✅ Có thể | Trung bình |
| __SERIALIZABLE__ | ❌ Ngăn | ❌ Ngăn | ❌ Ngăn | Chậm nhất |

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public void processPayment(Long orderId) {
    // ...
}
```

> __Best Practice__: Hầu hết trường hợp dùng __READ_COMMITTED__ (mặc định của hầu hết database). Chỉ dùng __SERIALIZABLE__ khi thực sự cần (ảnh hưởng performance).

## 5. Rollback Rules

### 5.1. Mặc định

Mặc định, `@Transactional` sẽ:
- __ROLLBACK__ khi gặp __unchecked exception__ (`RuntimeException` và subclass).
- __KHÔNG ROLLBACK__ khi gặp __checked exception__ (`Exception` và subclass, trừ `RuntimeException`).

```java
@Transactional
public void process() {
    // RuntimeException → ROLLBACK ✅
    throw new RuntimeException("Error");
}

@Transactional
public void process() throws IOException {
    // Checked Exception → KHÔNG ROLLBACK ❌
    throw new IOException("File error");
}
```

<br />

### 5.2. Tùy chỉnh Rollback

```java
// Rollback cho checked exception
@Transactional(rollbackFor = Exception.class)
public void process() throws Exception {
    throw new IOException("File error");  // SẼ ROLLBACK
}

// Không rollback cho exception cụ thể
@Transactional(noRollbackFor = EmailException.class)
public void createUserWithEmail(UserRequest request) {
    userRepository.save(user);           // Vẫn COMMIT
    emailService.sendWelcomeEmail(user); // EmailException → không rollback
}
```

## 6. Các lỗi thường gặp với `@Transactional`

### 6.1. Gọi method trong cùng class (Self-invocation)

```java
@Service
public class UserService {

    public void methodA() {
        // ❌ SAI: Gọi nội bộ → @Transactional KHÔNG hoạt động
        // Vì Spring AOP proxy không can thiệp được
        this.methodB();
    }

    @Transactional
    public void methodB() {
        // Transaction KHÔNG được tạo khi gọi từ methodA()
    }
}
```

__Giải pháp__: Tách method ra class khác, hoặc inject self reference.

```java
// Giải pháp 1: Tách class
@Service
public class UserService {
    @Autowired
    private UserTransactionService transactionService;

    public void methodA() {
        transactionService.methodB();  // ✅ Hoạt động
    }
}

@Service
public class UserTransactionService {
    @Transactional
    public void methodB() { ... }
}
```

<br />

### 6.2. `@Transactional` trên private method

```java
@Service
public class UserService {

    @Transactional  // ❌ KHÔNG hoạt động trên private method
    private void saveUser(User user) {
        userRepository.save(user);
    }
}
```

> __`@Transactional` chỉ hoạt động trên public methods__ vì Spring sử dụng __AOP proxy__ để quản lý transaction.

<br />

### 6.3. Nuốt Exception trong try-catch

```java
@Transactional
public void process() {
    try {
        // Thao tác database
        userRepository.save(user);
        throw new RuntimeException("Error");
    } catch (Exception e) {
        // ❌ SAI: Nuốt exception → Spring không biết để rollback
        log.error("Error occurred", e);
    }
}
```

__Giải pháp__: Re-throw exception hoặc gọi `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`.

## 7. Câu hỏi phỏng vấn thường gặp

__Q: `@Transactional(readOnly = true)` có tác dụng gì?__

Nó thông báo cho Hibernate rằng transaction chỉ đọc dữ liệu, __không cần flush changes__ về database → cải thiện performance. Hibernate cũng có thể bỏ qua __dirty checking__.

__Q: Tại sao `@Transactional` không hoạt động khi gọi nội bộ?__

Vì Spring dùng __AOP proxy__ để wrap bean. Khi gọi method nội bộ (`this.method()`), cuộc gọi đi trực tiếp đến đối tượng gốc, __không qua proxy__ → annotation bị bỏ qua.

__Q: REQUIRED vs REQUIRES_NEW?__

`REQUIRED` tham gia transaction hiện tại (hoặc tạo mới nếu chưa có). `REQUIRES_NEW` luôn tạo transaction mới, tạm treo transaction hiện tại. Dùng `REQUIRES_NEW` khi cần đảm bảo thao tác độc lập (ví dụ: audit logging).
