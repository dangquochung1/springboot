# Microservices và Spring Cloud

<br />

## 1. Microservices Architecture

### 1.1. Microservices là gì?

__Microservices__ là một kiến trúc phần mềm trong đó ứng dụng được chia thành nhiều __dịch vụ nhỏ, độc lập__, mỗi dịch vụ:
- Chạy trong __process riêng__.
- Giao tiếp qua __API__ (thường là REST hoặc messaging).
- Có thể __deploy, scale, update__ độc lập.
- Có __database riêng__ (Database per Service).

<br />

### 1.2. Monolith vs Microservices

| Tiêu chí | Monolith | Microservices |
| -------- | -------- | ------------- |
| __Cấu trúc__ | 1 ứng dụng lớn | Nhiều service nhỏ |
| __Deploy__ | Deploy cả ứng dụng | Deploy từng service |
| __Scale__ | Scale toàn bộ | Scale từng service |
| __Technology__ | 1 tech stack | Mỗi service có thể dùng tech khác nhau |
| __Database__ | 1 database chung | Database riêng mỗi service |
| __Team__ | 1 team lớn | Nhiều team nhỏ, mỗi team phụ trách 1-2 service |
| __Phức tạp__ | Code phức tạp, deploy đơn giản | Code đơn giản, deploy phức tạp |

<br />

### 1.3. Khi nào nên dùng Microservices?

__Nên dùng khi:__
- Ứng dụng lớn, phức tạp, nhiều team phát triển.
- Cần __scale__ các phần khác nhau một cách độc lập.
- Cần __deploy nhanh__ và thường xuyên.
- Các phần của ứng dụng có __load khác nhau__.

__Không nên dùng khi:__
- Ứng dụng nhỏ, team nhỏ (1-5 người).
- Chưa hiểu rõ domain (nên bắt đầu bằng monolith).
- Không có hạ tầng DevOps đủ mạnh.

## 2. Spring Cloud

### 2.1. Spring Cloud là gì?

__Spring Cloud__ là bộ công cụ giúp xây dựng các __distributed systems__ và __microservices__ trên nền tảng Spring Boot. Nó cung cấp giải pháp cho các vấn đề phổ biến trong kiến trúc microservices.

| Vấn đề | Giải pháp Spring Cloud |
| ------ | ---------------------- |
| __Service Discovery__ | Spring Cloud Netflix Eureka |
| __API Gateway__ | Spring Cloud Gateway |
| __Load Balancing__ | Spring Cloud LoadBalancer |
| __Circuit Breaker__ | Resilience4j |
| __Config Management__ | Spring Cloud Config |
| __Distributed Tracing__ | Micrometer Tracing (Zipkin) |
| __Messaging__ | Spring Cloud Stream (RabbitMQ, Kafka) |

<br />

## 3. Service Discovery (Eureka)

### 3.1. Service Discovery là gì?

Trong microservices, các service cần biết __địa chỉ__ của nhau để giao tiếp. __Service Discovery__ cho phép các service __tự đăng ký__ và __tìm kiếm__ nhau mà không cần hardcode URL.

<br />

### 3.2. Eureka Server

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-server</artifactId>
</dependency>
```

```java
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
```

```properties
server.port=8761
eureka.client.register-with-eureka=false
eureka.client.fetch-registry=false
```

<br />

### 3.3. Eureka Client (Service)

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
```

```properties
spring.application.name=user-service
server.port=8081
eureka.client.service-url.defaultZone=http://localhost:8761/eureka/
```

Gọi service khác qua __service name__ (không cần biết IP/port):
```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final RestClient.Builder restClientBuilder;

    public UserResponse getUser(Long userId) {
        return restClientBuilder.build()
                .get()
                .uri("http://user-service/api/users/{id}", userId)
                .retrieve()
                .body(UserResponse.class);
    }
}
```

## 4. API Gateway

### 4.1. API Gateway là gì?

__API Gateway__ là __điểm vào duy nhất__ (single entry point) cho tất cả requests từ client đến các microservices. Nó thực hiện:
- __Routing__: Chuyển request đến service phù hợp.
- __Load Balancing__: Phân tải giữa các instances.
- __Authentication__: Xác thực tập trung.
- __Rate Limiting__: Giới hạn số request.
- __Logging/Monitoring__: Ghi log tập trung.

<br />

### 4.2. Spring Cloud Gateway

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://USER-SERVICE
          predicates:
            - Path=/api/users/**
        - id: order-service
          uri: lb://ORDER-SERVICE
          predicates:
            - Path=/api/orders/**
        - id: product-service
          uri: lb://PRODUCT-SERVICE
          predicates:
            - Path=/api/products/**
```

## 5. Circuit Breaker (Resilience4j)

### 5.1. Circuit Breaker Pattern

__Circuit Breaker__ ngăn chặn __cascade failures__ khi một service bị lỗi. Nó có 3 trạng thái:

| Trạng thái | Mô tả |
| ---------- | ----- |
| __CLOSED__ | Bình thường, request đi qua |
| __OPEN__ | Service lỗi, trả về fallback ngay (không gọi service) |
| __HALF_OPEN__ | Cho phép một số request thử lại |

<br />

### 5.2. Sử dụng Resilience4j

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-circuitbreaker-resilience4j</artifactId>
</dependency>
```

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final RestClient restClient;

    @CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
    public UserResponse getUser(Long userId) {
        return restClient.get()
                .uri("http://user-service/api/users/{id}", userId)
                .retrieve()
                .body(UserResponse.class);
    }

    // Fallback method — được gọi khi circuit breaker OPEN
    public UserResponse getUserFallback(Long userId, Throwable throwable) {
        return new UserResponse(userId, "Unknown User", "N/A");
    }
}
```

```properties
# Circuit Breaker configuration
resilience4j.circuitbreaker.instances.userService.sliding-window-size=10
resilience4j.circuitbreaker.instances.userService.failure-rate-threshold=50
resilience4j.circuitbreaker.instances.userService.wait-duration-in-open-state=10s
```

## 6. Spring Cloud Config

### 6.1. Centralized Configuration

__Spring Cloud Config__ cho phép quản lý cấu hình của tất cả microservices từ __một nơi tập trung__ (Git repository hoặc file system).

Config Server:
```java
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}
```

```properties
server.port=8888
spring.cloud.config.server.git.uri=https://github.com/myorg/config-repo
```

Config Client:
```properties
spring.config.import=configserver:http://localhost:8888
spring.application.name=user-service
```

## 7. Spring WebFlux

### 7.1. Reactive Programming là gì?

__Reactive Programming__ là mô hình lập trình __non-blocking__, __asynchronous__, xử lý data streams. Thay vì chờ đợi (blocking), ứng dụng phản ứng (react) khi có dữ liệu.

| Spring MVC | Spring WebFlux |
| ---------- | -------------- |
| __Blocking I/O__ | __Non-blocking I/O__ |
| Thread-per-request | Event loop |
| `Servlet API` | `Reactive Streams API` |
| Trả về `Object` | Trả về `Mono<T>` hoặc `Flux<T>` |
| Phù hợp: CRUD, business apps | Phù hợp: Streaming, IoT, high-concurrency |

<br />

### 7.2. Mono và Flux

| Type | Mô tả |
| ---- | ----- |
| `Mono<T>` | 0 hoặc 1 phần tử (tương tự `Optional`) |
| `Flux<T>` | 0 đến N phần tử (tương tự `List/Stream`) |

```java
@RestController
@RequestMapping("/api/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    @GetMapping("/{id}")
    public Mono<User> getUser(@PathVariable Long id) {
        return userService.findById(id);
    }

    @GetMapping
    public Flux<User> getAllUsers() {
        return userService.findAll();
    }
}
```

## 9. Câu hỏi phỏng vấn thường gặp

__Q: Microservices giao tiếp với nhau như thế nào?__

__Synchronous__: REST API (RestClient, WebClient, Feign Client). __Asynchronous__: Message Queue (RabbitMQ, Apache Kafka). Nên ưu tiên asynchronous cho loose coupling.

__Q: Làm sao xử lý khi 1 service bị down?__

Sử dụng __Circuit Breaker__ (Resilience4j) để trả về fallback response. Kết hợp __Retry__ và __Rate Limiter__. Thiết kế __idempotent APIs__ và sử dụng __message queue__ cho eventual consistency.

__Q: Service Discovery hoạt động thế nào?__

Mỗi service khởi động → đăng ký với Eureka Server (tên + IP + port). Khi service A cần gọi service B → hỏi Eureka lấy địa chỉ B → gọi trực tiếp. Eureka heartbeat kiểm tra service còn sống không.

__Q: Spring MVC vs Spring WebFlux?__

Spring MVC: blocking, thread-per-request, phù hợp phần lớn ứng dụng. WebFlux: non-blocking, event-driven, phù hợp high-concurrency (streaming, IoT). Không nên dùng WebFlux nếu không thực sự cần.
